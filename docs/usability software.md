## 1. Оценка ПО по атрибутам Usability (ISO/IEC 25010:2011)

### 1.1 Распознаваемость соответствия *(Appropriateness Recognisability)*

> Способность пользователя распознать, соответствует ли продукт его потребностям.

На текущем этапе проекта реализован **backend с REST API**. UI отсутствует. Тем не менее атрибут частично покрыт:

- API доступно через **Swagger UI** (`/docs`) — разработчик и интегратор сразу видят все доступные ресурсы и их назначение.
- Все эндпоинты сгруппированы по тегам: `Users`, `Addresses`, `Products`, `Categories` — смысловое разделение сразу понятно.
- Имена ресурсов соответствуют предметной области: `/products`, `/orders`, `/users/{id}/addresses`.

**Оценка:** ✅ Удовлетворительно для API-first подхода. Для конечного пользователя требуется UI (запланирован в следующих спринтах).

***

### 1.2 Обучаемость *(Learnability)*

> Простота освоения продукта новыми пользователями.

- **Swagger UI** (`/docs`) автоматически генерируется FastAPI и содержит описание каждого endpoint, входные параметры и примеры ответов — нулевой порог для onboarding разработчика.
- Каждый сервис имеет `GET /health` — простейший «точка входа», который даёт понять, что сервис жив и готов к работе.
- API соответствует REST-конвенциям: `GET` — получить, `POST` — создать, `PATCH` — обновить, `DELETE` — удалить, что снижает когнитивную нагрузку.

**Было (до спринта 2):** эндпоинты не были задокументированы, структура проекта отсутствовала.

**Стало (после спринта 2):**
```
GET /docs         → Полная интерактивная документация API
GET /health       → Статус сервиса
GET /products     → Список с фильтрами (category_id, min_price, max_price, page, limit)
```

**Оценка:** ✅ Хорошо для backend. Запланировано: UI с онбордингом при регистрации.

***

### 1.3 Используемость / Операбельность *(Operability)*

> Удобство управления продуктом и его контроля.

Значительный вклад в операбельность внесён через **проектирование API**:

- **Пагинация** на `/products` с параметрами `page`, `limit`, поле `pages` в ответе — пользователь никогда не получает бесконечный список.
- **Фильтрация** по цене, категории, продавцу — сужение выборки без повторных запросов.
- **Soft delete** у пользователей и товаров (`is_active = false`) — операция обратима, пользователь не теряет данные необратимо.
- **Адрес по умолчанию** (`is_default`) — при оформлении заказа не нужно каждый раз выбирать адрес заново.
- **Атомарный сброс флага default** в репозитории — нельзя случайно поставить два дефолтных адреса.

**Оценка:** ✅ Хорошо. Операции предсказуемы и контролируемы.

***

### 1.4 Защита от ошибок пользователя *(User Error Protection)*

> Предотвращение ввода некорректных данных.

Реализовано несколько уровней защиты:

| Уровень | Реализация |
|---------|-----------|
| **Схема (Pydantic)** | `price: Decimal = Field(gt=0)`, `stock: int = Field(ge=0)`, `EmailStr` для email |
| **Бизнес-логика** | Проверка дублей (`409 Conflict`) при повторном `keycloak_id` или `slug` категории |
| **HTTP статусы** | `404` — ресурс не найден, `409` — конфликт, `400` — некорректный запрос |
| **Ownership check** | Метод `_get_owned()` в AddressService — нельзя изменить чужой адрес |

**Было:** валидация отсутствовала, любой запрос мог уйти в БД.

**Стало:**
```python
# Pydantic блокирует запрос ещё до попадания в сервис
price: Decimal = Field(gt=0, decimal_places=2)
stock: int = Field(ge=0)

# Сервис проверяет бизнес-правила
if await self.repo.get_by_keycloak_id(data.keycloak_id):
    raise HTTPException(status_code=409, detail="User already exists")
```

**Оценка:** ✅ Хорошо. Запланировано: валидация форм на UI (frontend).

***

### 1.5 Эстетика GUI *(User Interface Aesthetics)*

> Визуальная привлекательность и согласованность интерфейса.

На текущем этапе **GUI не реализован** — проект находится на стадии backend. Однако заложена основа:

- Swagger UI предоставляет чистый, структурированный интерфейс для работы с API в ходе разработки.
- Спроектированы схемы ответов: `UserPublicRead` (публичный, без чувствительных данных) и `UserRead` (полный) — будущий UI сможет отображать ровно то, что нужно, без лишней фильтрации на клиенте.
- `ProductListItem` — облегчённая схема для списков, содержит только нужные поля для карточки товара.

**Оценка:** ⏳ Запланировано. Wireframes и UI запланированы в следующих спринтах.

***

### 1.6 Доступность *(Accessibility)*

> Возможность использования продукта людьми с ограниченными возможностями (WCAG 2.0).

На уровне API:
- Все ответы возвращаются в **JSON** — любой assistive technology может работать с этим форматом.
- Разделены схемы `UserPublicRead` / `UserRead` — минимум данных для публичного API снижает нагрузку на интерпретацию.
- Запланировано: при разработке UI будут соблюдаться требования WCAG 2.0 (контрастность, aria-label, keyboard navigation).

**Оценка:** ⏳ Частично. Полная оценка возможна после реализации UI.

## 2. Выбранные пути улучшения UX и обоснование

Поскольку UI находится в планах, улучшения были сосредоточены на **UX уровня API** — это уровни **стратегии**, **набора возможностей** и **структуры** по модели Garrett.

### 2.1 Улучшение: Структурированные ошибки API

**Проблема (до):** FastAPI по умолчанию возвращает ошибки валидации в виде массива технических объектов, которые сложно интерпретировать клиенту.

**После:**
```json
// До (стандартная ошибка FastAPI)
{
  "detail": [
    {
      "loc": ["body", "price"],
      "msg": "ensure this value is greater than 0",
      "type": "value_error.number.not_gt"
    }
  ]
}

// После (человекочитаемый ответ)
{
  "detail": "Price must be greater than 0"
}
```

***

### 2.2 Улучшение: Пагинация с метаданными

**Проблема (до):** возврат «голого» списка товаров — клиент не знает, сколько страниц всего.

**После (`PaginatedProducts`):**
```json
{
  "items": [...],
  "total": 150,
  "page": 1,
  "limit": 20,
  "pages": 8
}
```
Клиент сразу знает: всего 150 товаров, 8 страниц → можно отрисовать пагинатор без дополнительных запросов.

***

### 2.3 Улучшение: Адрес по умолчанию

**Проблема (до):** пользователь должен каждый раз выбирать адрес при оформлении заказа.

**После:**
- `POST /users/{id}/addresses/{addr_id}/set-default` — один запрос выставляет адрес как основной.
- При создании нового адреса с `is_default: true` — предыдущий автоматически снимается.

***

### 2.4 Улучшение: Разделение публичного и приватного профиля

**Проблема (до):** один endpoint отдавал все данные пользователя, включая `keycloak_id`, `phone`, `is_active`.

**После:**
- `GET /users/{id}` — полный профиль (только для самого пользователя / admin).
- `GET /users/public/{id}` — публичный профиль (`first_name`, `last_name`, `avatar_key`, `is_seller`, `seller_profile`).
