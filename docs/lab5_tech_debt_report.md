# Лабораторная работа №5 — Анализ технического долга и план его устранения

## 1. Признаки технического долга в проекте

### 1.1 Непонятный / нечитабельный код
- **Результат анализа:** В целом качество кода удовлетворительное — именование переменных и функций последовательное, структура проекта понятна. Однако есть области, требующие доработки.
- **Примеры:**
  - Смешанные подходы к обработке ошибок: Python-сервисы и Go-сервисы используют разные паттерны для формирования HTTP-ответов об ошибках, что затрудняет единообразное восприятие кодовой базы.
  - Захардкоженные сообщения об ошибках разбросаны по разным файлам без централизации.
  - «Магические числа» в конфигурациях портов и других параметрах разнесены по Dockerfile и конфигурационным файлам без вынесения в переменные окружения или единый конфиг.
- **Влияние:** Среднее — непоследовательная обработка ошибок усложняет отладку, захардкоженные значения снижают поддерживаемость.

### 1.2 Дублирующийся код
- **Результат анализа:** Обнаружено значительное дублирование кода между сервисами, особенно в паттернах обработки ошибок и валидации входных данных.
- **Примеры:**
  - Паттерн обработки ошибок (формирование HTTP-ответа с кодом и описанием) повторяется многократно в Python-сервисах.
  - Аналогичный паттерн для Go-сервисов дублируется в обработчиках запросов.
  - Dockerfile-ы для Python-сервисов практически идентичны друг другу.
  - Логика подключения к базе данных и начальной конфигурации продублирована в нескольких сервисах.
- **Влияние:** Высокое — увеличивает затраты на поддержку, создаёт риск рассогласования и повышает вероятность ошибок при внесении изменений.

### 1.3 Отсутствие автоматизации (тесты, сборка, развёртывание)
- **Результат анализа:** Критическое отсутствие автоматизации по всему проекту.
- **Примеры:**
  - Тестовые файлы отсутствуют во всех сервисах.
  - Конфигурация CI/CD отсутствует (нет конфигурационных файлов для GitHub Actions, GitLab CI, Jenkins и т.п.).
  - Автоматизированных скриптов сборки, кроме базового Docker Compose, нет.
  - Автоматизированное развёртывание не настроено.
- **Влияние:** Критическое — высокий риск регрессий, ручной процесс деплоя, отсутствие качественных ворот (quality gates).

### 1.4 Запутанная архитектура / ненужные сложные зависимости
- **Результат анализа:** В целом микросервисная архитектура достаточно чистая, но имеются непоследовательности.
- **Примеры:**
  - Использование двух языков программирования (Python и Go) для разных сервисов без явного обоснования выбора.
  - Непоследовательная структура проекта между Python- и Go-сервисами.
  - В некоторых сервисах обнаружены дублирующиеся точки входа (несколько main-файлов).
  - Часть сервисов не использует паттерн Dependency Injection.
- **Влияние:** Среднее — повышает когнитивную нагрузку на разработчиков при переключении между сервисами.

### 1.5 Медленные / неэффективные средства
- **Результат анализа:** Выбор технологий в целом адекватный, но выявлены отдельные неэффективности.
- **Примеры:**
  - Множественные подключения к базе данных без оптимизации пула соединений (connection pooling).
  - Отсутствие слоя кэширования для часто запрашиваемых данных.
  - Синхронные файловые операции в асинхронных Python-сервисах (работа с объектным хранилищем).
  - Не обнаружена стратегия оптимизации запросов к базе данных или индексации.
- **Влияние:** Среднее — может повлиять на производительность при росте нагрузки.

### 1.6 Незакоммиченный код / долгоживущие ветки
- **Результат анализа:** Явных признаков незакоммиченного кода или долгоживущих веток не обнаружено.
- **Примеры:**
  - Комментарии TODO/FIXME/HACK в кодовой базе не найдены.
  - История Git выглядит чистой.
- **Влияние:** Низкое — непосредственных проблем не выявлено.

### 1.7 Отсутствие / несоответствие технической документации
- **Результат анализа:** Качество документации неоднородное.
- **Примеры:**
  - Файл README.md в корне проекта пуст или практически не содержит информации.
  - Имеется документация по архитектуре проекта в папке docs/.
  - Имеется документация по парному программированию.
  - Отсутствует документация по API и отдельным сервисам.
  - Нет руководства по развёртыванию и настройке среды разработки.
- **Влияние:** Среднее — новым разработчикам будет сложно разобраться в проекте и начать вносить вклад.

### 1.8 Отсутствие тестовой среды / непрерывной интеграции
- **Результат анализа:** Полное отсутствие тестовой инфраструктуры.
- **Примеры:**
  - Тестовые файлы и директории отсутствуют во всех сервисах.
  - Конфигурация CI/CD-пайплайна отсутствует.
  - В Docker Compose нет автоматизированного запуска тестов.
  - Конфигурация staging- или тестовой среды отсутствует.
- **Влияние:** Критическое — нет контроля качества, высокий риск проблем в продакшене.

## 2. План мероприятий по устранению долга

### 2.1 Действия по симптомам

**Симптом: Дублирующийся код**
- **Действия:**
  1. Создать общую библиотеку обработки ошибок для Python-сервисов.
  2. Создать общие утилиты обработки ошибок для Go-сервисов.
  3. Вынести общие паттерны Dockerfile в базовые образы.
  4. Создать общий модуль конфигурации подключения к БД.
- **Ожидаемый эффект:** Сокращение дублирования кода на ~60%, повышение единообразия, упрощение поддержки.

**Симптом: Отсутствие автоматизации**
- **Действия:**
  1. Реализовать unit-тесты для всех сервисов (целевое покрытие — 80%).
  2. Настроить CI/CD-пайплайн (GitHub Actions или аналог).
  3. Добавить интеграционные тесты для взаимодействия сервисов.
  4. Реализовать автоматизированные скрипты развёртывания.
- **Ожидаемый эффект:** Повышение качества кода, обеспечение непрерывной доставки, снижение количества ручных ошибок.

**Симптом: Проблемы с документацией**
- **Действия:**
  1. Создать полноценный README проекта.
  2. Добавить документацию API с использованием OpenAPI/Swagger.
  3. Написать руководство по настройке среды разработки.
  4. Задокументировать процедуры развёртывания.
- **Ожидаемый эффект:** Улучшение онбординга разработчиков, уменьшение зависимости от «знаний в головах».

**Симптом: Непоследовательность архитектуры**
- **Действия:**
  1. Стандартизировать структуру проекта для всех сервисов.
  2. Удалить дублирующиеся точки входа.
  3. Внедрить единообразный паттерн Dependency Injection.
  4. Задокументировать обоснование выбора технологий и план миграции.
- **Ожидаемый эффект:** Снижение когнитивной нагрузки, повышение поддерживаемости.

- **Затронутые сервисы:** Все, особенно сервисы заказов, оплаты, пользователей и каталога.

**Группа 3: Документация и онбординг**

## 4. Сравнение технического долга и недоимплементированных фич

**Список ключевых отсутствующих или недореализованных функций:**
- API Gateway для единой точки входа
- Middleware аутентификации/авторизации (помимо базовой интеграции с Keycloak)
- Rate limiting и throttling
- Мониторинг и наблюдаемость (логи, метрики, трассировка)
- Слой кэширования
- Функция поиска
- Рекомендательная система
- Аналитика и отчётность
- Шаблоны email/SMS-уведомлений
- Панель администратора
- Отзывы и рейтинги товаров
- Персистентная корзина
- Система отслеживания заказов
- Интеграция с платёжными провайдерами
- Управление запасами
- Мультиязычность

**Сравнение и обоснование:**
- **Объём технического долга:** Большой — критические проблемы в области тестирования, CI/CD и дублирования кода.
- **Объём недореализованных фич:** Очень большой — множество ключевых функций e-commerce ещё не реализовано.
- **Оценка:** Объём недореализованных фич превышает объём технического долга, однако технический долг более критичен для устойчивости проекта. Отсутствие CI/CD и тестов замедляет реализацию любых новых фич и увеличивает риск регрессий.

**Анализ краткосрочной ценности:**
- **Снижение технического долга** даёт немедленную отдачу: рост продуктивности разработчиков, уменьшение количества багов, ускорение циклов доставки.
- **Реализация фич** приносит пользовательскую ценность, но при наличии технического долга — с повышенными затратами на поддержку.
- **Рекомендация:** Сначала устранить критический технический долг, затем реализовывать фичи инкрементально.

## 5. Выводы и корректировка плана разработки

**Необходимость и приоритетность устранения долга:**
- **Критический:** Тестовая инфраструктура и CI/CD-пайплайн — необходимо устранить немедленно.
- **Высокий:** Дублирование кода и стандартизация обработки ошибок — устранить в следующей итерации.
- **Средний:** Доработка документации и стандартизация архитектуры — устранить в течение 2–3 итераций.
- **Низкий:** Оптимизация производительности — можно отложить до момента, когда нагрузочное тестирование выявит проблемы.

**Критические элементы долга для ближайших итераций:**
1. **Тестовая инфраструктура** — без тестов любая разработка фич сопряжена с высоким риском.
2. **CI/CD-пайплайн** — ручное развёртывание не масштабируется.
3. **Общая обработка ошибок** — ускорит всю последующую разработку.
4. **Базовая документация** — необходима для командной работы.

**Элементы, которые можно отложить:**
- Оптимизация производительности (connection pooling, кэширование).
- Расширенный мониторинг и наблюдаемость.
- Стандартизация архитектуры между Python- и Go-сервисами.
- Улучшение единообразия стиля кода.

**Обновлённый план разработки на ближайшие итерации:**

**Итерация 1 (2 недели):**
- Неделя 1: Внедрить тестовый фреймворк и написать тесты для 2 ключевых сервисов (пользователи, каталог).
- Неделя 2: Настроить базовый CI/CD-пайплайн и создать общие утилиты обработки ошибок.

**Итерация 2 (2 недели):**
- Неделя 1: Завершить покрытие тестами оставшихся сервисов и добавить интеграционные тесты.
- Неделя 2: Создать полноценную документацию и документацию API.

**Итерация 3 (2 недели):**
- Неделя 1: Провести рефакторинг дублирующегося кода и стандартизировать структуру проекта.
- Неделя 2: Реализовать 1–2 высокоприоритетные фичи с использованием улучшенной инфраструктуры.

**Интеграция устранения долга в Definition of Done:**
- Код должен содержать unit-тесты с минимальным покрытием 80%.
- Код должен проходить CI/CD-пайплайн, включая автоматизированные тесты.
- Код должен использовать общие паттерны обработки ошибок.
- Новые фичи должны сопровождаться документацией API.
- Код должен соответствовать стилевым рекомендациям проекта.
- Все сервисы должны развёртываться через автоматизированный пайплайн.
- Для значимых изменений должна быть проведена оценка влияния на производительность.

**Метрики успеха:**
- Покрытие тестами >80% для всех сервисов.
- Полное отсутствие ручных шагов при развёртывании.
- Доля дублирования кода <5%.
- Полная документация API для всех эндпоинтов.
- Время онбординга нового разработчика <4 часов.
- Время выполнения CI/CD-пайплайна <10 минут.
