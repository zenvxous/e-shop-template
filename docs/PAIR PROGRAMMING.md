# Парное программирование

## Задача

В рамках второго спринта проекта электронного магазина мы вдвоём реализовали **два микросервиса на FastAPI**:

- **Catalog Service (порт 8002)** — управление товарами, категориями и изображениями товаров. Модели: `Category` (иерархия через self-referencing FK), `Product`, `ProductImage`. API включает CRUD товаров с фильтрацией и пагинацией, построение дерева категорий, управление изображениями.
- **User Service (порт 8001)** — управление пользователями, профилями продавцов и адресами доставки. Модели: `User`, `SellerProfile` (one-to-one), `Address`. API включает синхронизацию с Keycloak, загрузку аватаров в MinIO, механизм become-seller, CRUD адресов с флагом `is_default`, soft delete пользователей.

Оба сервиса построены по принципу Clean Architecture (Repository → Service → Router), используют async SQLAlchemy и Alembic для миграций. Параллельно велась работа над инфраструктурой: настройка MinIO (init-buckets.sh), Keycloak, RabbitMQ и Docker Compose с healthcheck'ами и разделением сетей.

Во время разработки мы попробовали два стиля парного программирования: **«на равных»** и **«пинг-понг»**.

---

## Опробованные стили

### На равных

Оба участника поочерёдно занимали роль «водителя». Смена происходила после завершения логического блока — например, после реализации слоя репозитория для `Product` или настройки эндпоинтов синхронизации с Keycloak в User Service. Архитектурные решения (структура слоёв, разделение ответственности между сервисами, формат Pydantic-схем) обсуждались совместно.

Этот стиль оказался **более комфортным и продуктивным**. Переключение ролей было естественным, и оба участника оставались вовлечены в процесс на протяжении всей сессии.

### Пинг-понг

Использовали при реализации отдельных эндпоинтов в формате TDD: один писал тесты (например, для CRUD адресов доставки или фильтрации товаров), второй — реализацию под них, затем роли менялись.

На практике стиль оказался **менее удобным**: постоянное переключение между тестами и кодом замедляло темп. Требовалось дополнительное время на согласование структуры тестов и строгое соблюдение очередности, к чему мы не были привычны.

---

## Что получилось хорошо

- **Качество кода.** Ошибки в валидации запросов, некорректные HTTP-статусы и проблемы с async-сессиями SQLAlchemy ловились сразу при написании, а не на этапе отладки через Swagger.
- **Обмен знаниями.** Делились опытом по FastAPI (Depends, BackgroundTasks, APIRouter), настройке Alembic для async, работе с MinIO SDK и конфигурации Pydantic Settings.
- **Продуманная архитектура.** Совместное обсуждение помогло правильно разделить ответственность между Catalog и User Service — например, решение хранить `seller_id` в catalog_db как UUID без cross-DB FK, а связь между сервисами делать только через HTTP.
- **Меньше отвлечений.** Работая в паре, реже переключались на посторонние вещи — время двоих ощущалось ценнее.
- **Общее понимание системы.** Оба участника одинаково хорошо разобрались в устройстве обоих микросервисов, инфраструктуре Docker Compose и принципах взаимодействия компонентов (Keycloak, MinIO, PostgreSQL, RabbitMQ).

---

## Трудности и их решения

| Проблема | Как проявлялась | Решение |
|---|---|---|
| **Разный темп работы** | Один участник быстрее ориентировался в async SQLAlchemy и структуре слоёв, второй не всегда успевал следить за ходом мысли | Делить работу на мелкие подзадачи (один эндпоинт / один слой за раз), делать паузы для синхронизации перед переходом к следующему блоку |
| **Различия в стиле кода** | Разные привычки в именовании переменных, организации импортов и структуре модулей | Договориться о единых правилах до начала работы, использовать black/ruff для автоформатирования и pre-commit хуки |
| **Пассивность при «пинг-понге»** | Участник, ожидающий своей очереди, терял вовлечённость и становился наблюдателем | Комментировать свои действия вслух, проговаривать текущую цель («сейчас пишу тест на создание категории с parent_id»), задавать вопросы по каждому шагу |

---

## Когда стоит применять ПП в будущем

Парное программирование оправдано для задач, где цена ошибки высока или важно общее понимание решения:

- проектирование архитектуры новых микросервисов и их API-контрактов;
- реализация сложной бизнес-логики (корзина, платежи, события через RabbitMQ);
- настройка инфраструктуры (Docker Compose, healthcheck'и, сети, init-контейнеры);
- рефакторинг и декомпозиция существующего кода;
- отладка сложных багов в асинхронном коде или при взаимодействии сервисов.

Для простых рутинных задач (шаблонные CRUD без особой логики, мелкие правки переменных окружения) индивидуальная работа эффективнее.

---

## Итог

Парное программирование при разработке Catalog Service и User Service на FastAPI показало себя как рабочий инструмент командной разработки. Стиль «на равных» подошёл лучше — он сохраняет вовлечённость обоих участников и позволяет гибко переключаться между задачами. Подход заметно улучшил качество кода, ускорил обмен знаниями по стеку (FastAPI, async SQLAlchemy, MinIO, Keycloak) и помог сформировать общее понимание архитектуры проекта.
